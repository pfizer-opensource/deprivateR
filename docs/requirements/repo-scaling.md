# Requirements: Repo Scaling

Use this document to capture the requirements for restructuring
`evgen-open-data` so it can host many datasets (and more contributors)
without the per-dataset friction we saw with the `pop-est` POC.

This is a *brief* — the implementer (human or AI) should not start work
until each Required field below is filled in. Anything marked
*Optional* can be left blank or "TBD".

The companion to this doc is `docs/requirements/_dataset-template.md`, which
governs **per-dataset** specs. This doc governs the **repo itself**.

---

## 1. Overview

- **Goal in one sentence** (Required): Introduce the concept of "data packages" and reorganize the repo around this idea.
- **Why now?** (Required — what about adding the next 1–N datasets is
  painful or risky under the current layout?): I want to make sure that code, tests, and documentation for each dataset are neatly encapsulated, so adding a new dataset doesn't involve touching files from other datasets or the shared functions.
- **In scope** (Required — bullets):
  - Restructure the folder layout to have `datasets/<dataset-slug>/` for each dataset's pipeline.
  - Move cross-dataset helpers from `source/functions/` to `source/` (the top-level `source/functions/` directory is deleted) and ensure they don't reference specific datasets.
  - Define a clear contract for what files go in each dataset folder and what goes in the shared helpers.
  - Update documentation and tests to reflect the new structure.
  - Migrate the existing `pop-est` dataset to the new layout.
  - Refactor existing helpers so paths are computed from a "dataset root" (`datasets/<slug>/`) rather than hard-coded.
- **Out of scope** (Required — bullets; e.g. "no CI changes",
  "no package-ification"): None

## 2. Success Criteria

What must be true after this work lands? Phrase each as something a
reviewer can check.

- [ ] *The repo remains well-organized and navigable with 1–N datasets added* (Required — e.g. "no file in `source/` is dataset-specific", "documentation for any dataset is regenerated by a single command", "a contributor can find the canonical place for a new helper without asking").
- [ ] *Pan-dataset resources, such as `admin` and `source` reference data, are clearly separated from per-dataset pipelines* (Required — e.g. "no file in `source/` references a dataset slug", `aggregations/admin/` and `aggregations/source/` are built from reference data in the other datasets).
- [ ] *Datasets have their own independent QC pipelines and documentation that don't require touching other datasets' files* (Required — e.g. "a new dataset can be added by copying a single template folder and filling in <N> files").
- [ ] *Datasets can be rebuilt independent of all other datasets (except `aggregations/admin/` and `aggregations/source/`)*

### 3.2 Per-dataset folder convention (Required)

Define the canonical contents of `datasets/<dataset-slug>/source/`. This is the
contract a new dataset author must follow.

| File / folder      | Required? | Purpose |
|--------------------|-----------|---------|
| `<dataset-slug>.R` (entry, lives at `datasets/<dataset-slug>/source/<dataset-slug>.R`) | Yes | Main script that triggers all workflow steps to rebuild the dataset. |
| `workflow/<dataset-slug>-NN-<step>.R` | Yes | Stepwise workflow for building the data product. |
| `specs/<dataset-slug>-sources.R` | Yes | Single source of truth for vintage, citation, URL, format, and download targets. |
| `specs/<dataset-slug>-tables.R`  | Yes | Structured spec used by `render_dataset_doc()` to render per-table metadata, column dictionaries, and key relationships in `datasets/<dataset-slug>/docs/metadata/<dataset-slug>.md`. |
| `specs/<dataset-slug>-update.R`  | No (but recommended) | Script to check whether a new vintage has been released. |
| `skills/<dataset-slug>-update.md` | No (but recommended if a script is not feasible) | Skill file to check source websites manually; should include instructions for how to check for updates and a checklist of what to update when a new vintage drops. |
| `skills/<dataset-slug>-qc.md` | No (but recommended) | Dataset-specific QC skill, e.g. to run the §7 checks in the dataset spec. |
| `functions/` (folder, possibly empty) | No (folder may be empty) | Dataset-private helpers prefixed per §4.4. Helpers used by ≥2 datasets must live in top-level `source/` instead. |
| `datasets/<dataset-slug>/tests/test-<dataset-slug>.R` | Yes | Dataset-specific tests, e.g. for the §7 checks in the dataset spec. The `test-` prefix is required for `testthat::test_dir()` discovery. |

- **Slug-prefixed filenames are intentional**: even though the dataset folder already disambiguates by slug, prefixing every spec/workflow/skill/test file with the slug enables cross-repo grep (e.g. `rg pop-est-`), unambiguous tab completion when the user opens the workflow root, and copy-pasted error messages that always identify the dataset.
- **Naming convention for step files**: `<dataset-slug>-NN-<verb_noun>.R`, with `NN` zero-padded (e.g. `pop-est-01-raw_cache.R`).
  - **Phase numbers are labels**, not unique IDs. Multiple step files may share the same `NN` when they belong to the same logical phase (e.g., `pop-est-03-build_national.R`, `pop-est-03-build_region.R`, `pop-est-03-build_state.R`, `pop-est-03-build_county.R`). Sibling files within a phase are conventionally ordered alphabetically by their `<verb_noun>` suffix.
  - **Phase numbers must be nondecreasing** when discovered/sorted by `find_step_files()`. Gaps (e.g., 01, 03, 05) are permitted to give authors room to insert phases later without renumber churn.
  - **Run order is discovery order** — the entry script should iterate over `find_step_files()` results rather than hand-rolling its own sequence. (Pop-est currently violates this; tracked in #41.)
- **Where does the entry-point script live?** At `datasets/<dataset-slug>/source/<dataset-slug>.R`.

### 3.3 Per-dataset documentation layout (Required)

- Where does the dataset spec live?: `datasets/<dataset-slug>/docs/requirements/<dataset-slug>.md`
- Where does the generated dictionary live?: `datasets/<dataset-slug>/docs/metadata/<dataset-slug>.md`
- Where do PDF / other rendered exports live?: `datasets/<dataset-slug>/docs/metadata/exports/`
- Should there be a top-level `docs/datasets.md` index of all
  published datasets? **Yes — generated, not hand-maintained; implementation deferred to #35.** A generated index avoids the drift that motivated the original "no" answer.

### 3.4 Tests (Required)

- Where do unit tests for shared helpers (top-level `source/`) live?: `tests/`
- Where do per-dataset acceptance tests live (the §7 checks from each
  dataset spec)?: `datasets/<dataset-slug>/tests/`
- What runs them? `testthat::test_dir(...)` in a script.

### 3.5 Raw cache & outputs (Required)

- Confirm `datasets/<dataset-slug>/data/data_raw/` (gitignored) for raw inputs. — **CONFIRMED**
- Confirm `datasets/<dataset-slug>/data/<dataset-slug>-*.parquet` (and optional `datasets/<dataset-slug>/data/csv/` mirror) for outputs. — **CONFIRMED**
- Should outputs be committed, gitignored, or released as artifacts
  (e.g. GitHub Releases)? **gitignored**
- **Gitignore pattern**: replace v1 patterns (`data/*.parquet`, `data/csv/`, `data/data_raw/*/`) with `datasets/*/data/` (gitignores everything under each dataset's `data/`, including the raw cache). The `.gitkeep` convention may be used to preserve empty `datasets/<slug>/data/data_raw/` and `datasets/<slug>/data/csv/` placeholders.

### 3.6 `admin` and `source` aggregations (Required)

`admin` and `source` are not regular datasets — they are *aggregation
packages* that collect and deduplicate per-dataset `admin` and `source`
tables. They live in a separate top-level `aggregations/` peer (not
inside `datasets/`) so the dependency direction is explicit:
`aggregations/` reads from `datasets/`, never the other way around.

**Layout**: `aggregations/admin/` and `aggregations/source/` follow the
standard per-dataset folder convention defined in §3.2 (with their own
`source/`, `tests/`, `docs/`, and `data/` subfolders).

**Build trigger**: aggregations are **not** rebuilt on every dataset
rebuild. During development of a new or changed dataset, the aggregations
remain frozen so in-flight changes can't contaminate the canonical
reference tables. A dedicated **release skill** (`release_dataset/SKILL.md`,
§5.1) is the single entry point that, once a dataset has passed UAT,
registers it (or refreshes its registry entry) and triggers the
`aggregations/admin/` and `aggregations/source/` rebuilds in one shot.

**Dataset registry**: per-dataset manifests at
`datasets/<slug>/RELEASE.json` are the **source of truth**. The
top-level index at `data/dataset-registry.json` is a **derived
artefact**, regenerated deterministically from the manifests by
`register_dataset()` / `unregister_dataset()` (and by
`generate_dataset_registry_json()` in CI). Schema starts minimal —
initially just the slug per entry — and may grow over time (e.g.
last-released vintage, output paths) without a breaking change.
**Writes are helper-only**: `register_dataset()` / `unregister_dataset()`
(§4.2) are the only sanctioned ways to mutate manifests OR the derived
index; manual edits are discouraged and should fail review. The registry
is populated at **release time** (by `release_dataset/SKILL.md`), not at
dataset creation time, so unreleased scaffolds don't appear. Per-dataset
manifests eliminate the merge-conflict hotspot the global JSON would
otherwise become as soon as two dataset PRs are open simultaneously
(#26). Aggregation builds read the registry (via `read_dataset_registry()`)
to discover which datasets to pull from.

**Name-collision handling**: `source_id` is **namespaced at aggregation
time** (#27, Option A) as `<slug>:<bare-source-id>`. Dataset-local
source parquets keep bare ids (so per-dataset fact-table FKs remain
stable); `source-01-collect.R` prepends the slug when union-ing across
datasets. This makes cross-dataset collisions structurally impossible.
The collision check in `source-02-validate.R` is retained as
defense-in-depth: a non-empty conflict list now means a code bug (the
namespacing prefix was bypassed) rather than an author coordination
problem.

**Repo-level `data/` purpose**: reserved for cross-dataset artifacts
that are not associated with any single dataset. The derived registry
index (`data/dataset-registry.json`) is the first such artifact (the
canonical source-of-truth manifests live under each dataset, per #26).
This folder is **not** gitignored.

### 4.2 New helpers needed (Required)

#### `create_dataset_skeleton(slug)`

- **Signature**: `create_dataset_skeleton(slug, root = ".")`
- **Purpose**: scaffold a complete `datasets/<slug>/` package with all
  required subfolders and stub files, so `new_dataset/SKILL.md` is a
  thin wrapper around this function.
- **Inputs**: `slug` (kebab-case dataset slug); `root` (repo root,
  defaults to working directory).
- **Behaviour**:
  - Creates `datasets/<slug>/{data/data_raw,docs/{qc-reports,metadata/exports,requirements},source/{functions,skills,specs,workflow},tests}/` with `.gitkeep` placeholders where needed.
  - Writes scaffolded boilerplate (not empty stubs):
    - `datasets/<slug>/source/<slug>.R` — entry script with `library()` calls, `source()` of the helpers, ordered `with_step()` calls discovered via `find_step_files(slug)`.
    - `datasets/<slug>/source/specs/<slug>-sources.R` — empty `POP_EST_SOURCES`-style list with placeholder fields.
    - `datasets/<slug>/source/specs/<slug>-tables.R` — empty table-spec list.
    - `datasets/<slug>/docs/requirements/<slug>.md` — copy of `docs/requirements/_dataset-template.md`.
  - Appends `datasets/<slug>/data/` to `.gitignore` if not already covered by the wildcard pattern.
  - Does **not** touch `data/dataset-registry.json`; registry insertion happens at release time via `release_dataset/SKILL.md` (§5.1, §3.6).
- **Errors**: aborts if `datasets/<slug>/` already exists; aborts if `slug` is not valid kebab-case (must start with a lowercase letter and contain only lowercase letters, digits, and hyphens — see §9 / §4.4).
- **Tests** (Required): (1) happy path creates all expected paths; (2) re-running on an existing slug aborts cleanly without partial writes; (3) invalid slugs (uppercase, underscores, leading digits) are rejected.

#### Additional helpers (Required — please confirm or strike each)

These were proposed during review of v1 and are very likely needed to
keep the new layout DRY:

- **`dataset_root(slug, ...)` / `dataset_path(slug, kind, ...)`** — single source of truth for path computation. Every other helper takes a slug and uses these instead of hard-coding `datasets/<slug>/...`. `kind` is one of `"data"`, `"data_raw"`, `"csv"`, `"docs"`, `"metadata"`, `"requirements"`, `"qc-reports"`, `"functions"`, `"skills"`, `"specs"`, `"workflow"`, `"tests"`.
- **`find_step_files(slug)`** — return the sorted list of `<slug>-NN-*.R` files under `datasets/<slug>/source/workflow/`. Used by entry scripts and by `create_dataset_skeleton()` to wire the entry script.
- **`load_dataset_spec(slug)`** — `source()` `<slug>-sources.R` and `<slug>-tables.R` together (and validate them against the schemas defined in §3.2 of any per-dataset spec).
- **`register_dataset(slug, ...)` / `unregister_dataset(slug)`** — write or remove `datasets/<slug>/RELEASE.json` (the per-dataset manifest, source of truth) AND regenerate the derived `data/dataset-registry.json` index in the same call so the working tree is never inconsistent. `register_dataset()` is called from `release_dataset/SKILL.md` (§5.1) once a dataset has passed UAT; `unregister_dataset()` is called from the dataset-removal workflow. These are the **only** sanctioned writers of the registry artifacts (§3.6). `generate_dataset_registry_json()` regenerates the index from manifests for CI verification.
- **`render_qc_report(slug, results, run_id = NULL)`** — write a markdown QC report into `datasets/<slug>/docs/qc-reports/<run_id>.md`. Used by the §5 QC skills.

Each helper above must follow the helper-block template (signature,
inputs, errors, tests) when implemented; that level of detail is
deferred to implementation time once the design has been confirmed.

### 4.3 Helpers to refactor or rename (Required)

Almost every existing helper hard-codes v1 paths
(`data/data_raw/<slug>/`, `data/<slug>-*.parquet`,
`docs/datasets/<slug>.md`). The new layout breaks all of them. Each
helper below must be refactored to take a `slug` argument and resolve
its paths via `dataset_path(slug, kind)` (see §4.2). After refactor,
no helper in top-level `source/` may contain a literal `data/...`,
`docs/...`, or dataset-name string.

### 4.4 Function-library conventions (Required)

These rules apply to **all** R helpers in the repo, with one
location-specific addition.

**Inherited from v1 (unchanged):**

- Roxygen-style headers with `@param`, `@return`, and an `@examples`
  block where useful.
- snake_case names; verb-led (`build_admin_table`, not
  `admin_table_builder`).
- Package allow-list: `dplyr`, `tidyr`, `tibble`, `readr`, `readxl`,
  `arrow`, `cli`, `lifecycle`. Anything else requires a §9 open-questions entry.
- Side-effect helpers must accept `quiet = TRUE` to suppress
  `cli::cli_*` output.
- Helper signatures are stable within a major release; deprecations
  use `lifecycle::deprecate_warn()` (or equivalent) for one minor
  release before removal.

**New for v2 — naming prefix and visibility:**

- Helpers in **top-level `source/`** (cross-dataset reusables): **no
  naming prefix**. These are the public, documented surface area used
  by every dataset.
- Helpers in **`datasets/<slug>/source/functions/`** (dataset-private):
  must be prefixed with `<slug_underscored>_` (e.g.
  `pop_est_pivot_year_cols_long()` — slug `pop-est` becomes
  `pop_est_`). The prefix prevents accidental shadowing if a dataset
  helper is later promoted to top-level `source/`.
- All dataset-private helpers must carry `@keywords internal` in their
  roxygen block. This makes "this is dataset-specific, don't reuse it
  from elsewhere" visible at the top of the file.
- A helper graduating from dataset-private to cross-dataset is renamed
  (drop the prefix) and moved as a single commit, with a deprecation
  shim left behind for one release.
- The slug-prefix convention applies to **filenames only**, not to
  R-internal symbol names. Functions and constants defined inside a
  dataset's `specs/` files continue to use the v1 `POP_EST_*`
  UPPER_SNAKE convention (e.g. `<SLUG_UPPER>_SOURCES`,
  `<SLUG_UPPER>_TABLES`). This keeps spec files readable and avoids
  redundant prefixing on symbols that are already namespaced by the
  file they live in.

### 5.1 New skills needed (Required)

The skills below are **cross-cutting** (they apply to *any* dataset), so
they live in `.github/skills/`. The dataset-specific skill files they
*produce* (e.g. `<slug>-qc.md`, `<slug>-update.md`) live in
`datasets/<slug>/source/skills/`.

#### Create a New Dataset

- **Filename** (`<topic>_skill.md`): **`new_dataset/SKILL.md`**
- **Trigger** (when should the AI invoke it? Required keyword(s) /
  intent): When prompted to create a new dataset workflow
- **Inputs the user must supply**:
  - Dataset name and slug
  - Source information (URL, vintage, format, citation)
  - Any dataset-specific parameters (e.g. geographic level, frequency)
- **Steps the skill performs** (numbered):
  1. Create a new folder in `datasets/<dataset-slug>/` with the required subfolders (`data/`, `docs/`, `source/`, `tests/`).
  2. Scaffold the main entry script `datasets/<dataset-slug>/source/<dataset-slug>.R` with boilerplate to run the workflow.
  3. Point user to requirements document to fill out the dataset spec and workflow steps.
- **Outputs / artifacts produced**: the scaffolded folder and files for the new dataset, with placeholders filled in where possible.
- **Success criteria** (how the skill knows it's done): The new dataset folder is created with the correct structure, and the main entry script contains the necessary boilerplate to run the workflow.
- **Out of scope**: Running the workflow — it may need more customization before it is ready for production.

#### Invoke a Dataset-Specific Skill

- **Filename** (`<topic>_skill.md`): **`invoke_dataset/SKILL.md`**
- **Trigger** (when should the AI invoke it? Required keyword(s) /
  intent): When prompted to run a dataset-specific skill.
- **Inputs the user must supply**:
  - Dataset name and slug
  - Skill description
- **Steps the skill performs** (numbered): Search `datasets/<dataset-slug>/source/skills/` for a skill matching the description, and if found, follow that skill's instructions.
- **Outputs / artifacts produced**: Will be based on the specific skill called.
- **Success criteria** (how the skill knows it's done): The dataset-specific skill executes successfully. No other dataset skills should be called.
- **Out of scope**: Running the dataset pipeline itself. This skill only dispatches another skill from `datasets/<slug>/source/skills/`; pipeline runs are handled by the entry-point script (`datasets/<slug>/source/<slug>.R`).

#### Create Dataset-Specific QC Scaffolding

- **Filename** (`<topic>_skill.md`): **`create_dataset_qc/SKILL.md`**
- **Trigger**: When prompted to create a new test suite for a dataset.
- **Inputs the user must supply**: Dataset name and slug.
- **Steps the skill performs** (numbered):
  1. Create new acceptance-test files in `datasets/<dataset-slug>/tests/` with boilerplate for the §7 dataset-spec checks.
  2. Create new regression-test files in `datasets/<dataset-slug>/tests/` tailored to the specific dataset (cross-vintage drift, row-count stability, schema stability).
  3. Create a new dataset-specific QC skill at `datasets/<dataset-slug>/source/skills/<dataset-slug>-qc.md` that calls the test files and contains dataset-specific rules (vintage nuances, drift tolerances).
  4. Wire the regression test into the end of the dataset's workflow (as the highest-`NN` step file, e.g. `<slug>-99-regression.R`) so it runs as part of every build.
- **Outputs / artifacts produced**: scaffolded test files and `<slug>-qc.md` skill, with placeholders filled in where possible.
- **Success criteria**: The new test files are created with the correct structure; `<slug>-qc.md` is set up to invoke them; the regression test is appended to the workflow and runs successfully on the next build.
- **Out of scope**: Executing the QC checks (handled by `run_dataset_qc/SKILL.md`).

#### Run Dataset-Specific QC Checks

- **Filename** (`<topic>_skill.md`): **`run_dataset_qc/SKILL.md`**
- **Trigger**: When prompted to run an existing QC suite for a dataset.
- **Inputs the user must supply**: Dataset name and slug.
- **Steps the skill performs** (numbered):
  1. Locate `datasets/<dataset-slug>/source/skills/<dataset-slug>-qc.md` and follow its instructions.
  2. Execute the test files under `datasets/<dataset-slug>/tests/` via `testthat::test_dir()`.
  3. Render results to `datasets/<dataset-slug>/docs/qc-reports/<run_id>.md` via `render_qc_report()` (see §4.2 for `<run_id>` semantics).
- **Outputs / artifacts produced**: a markdown QC report committed under `datasets/<slug>/docs/qc-reports/`; a non-zero exit code on test failure.
- **Success criteria**: All tests pass; the QC report is written and committed.
- **Out of scope**: Creating the QC scaffolding (handled by `create_dataset_qc/SKILL.md`).

#### Create Dataset-Specific Update Check

- **Filename** (`<topic>_skill.md`): **`create_dataset_update/SKILL.md`**
- **Trigger**: When prompted to create a new update check for a dataset.
- **Inputs the user must supply**: Dataset name and slug.
- **Steps the skill performs** (numbered):
  1. Create a new script at `datasets/<dataset-slug>/source/specs/<dataset-slug>-update.R` with boilerplate for checking for vintage updates against the dataset's sources.
  2. Create a new skill at `datasets/<dataset-slug>/source/skills/<dataset-slug>-update.md` that calls the update check script and contains instructions for manual review when the script cannot be automated (e.g. a vintage page with no stable URL).
- **Outputs / artifacts produced**: the scaffolded update check script and skill, with placeholders filled in where possible.
- **Success criteria**: The new update check script is created with the correct structure, and the dataset-specific update skill is set up to run that script or provide instructions for manual checks.
- **Out of scope**: Executing the update check (handled by `run_dataset_update/SKILL.md`).

#### Run Dataset-Specific Update Check

- **Filename** (`<topic>_skill.md`): **`run_dataset_update/SKILL.md`**
- **Trigger**: When prompted to check whether a new vintage of a dataset is available, or as part of a scheduled run across all datasets.
- **Inputs the user must supply**: Dataset name and slug (or `"all"` to iterate over the dataset registry).
- **Steps the skill performs** (numbered):
  1. Locate `datasets/<dataset-slug>/source/skills/<dataset-slug>-update.md` and follow its instructions.
  2. If an update is detected, open a backlog issue (via `backlog/SKILL.md`) describing the new vintage and the files to refresh.
- **Outputs / artifacts produced**: a backlog issue per dataset that has a new vintage; no issue if nothing changed.
- **Success criteria**: Every dataset in the registry was checked; new vintages have backlog issues; no false positives on previously-acknowledged vintages.
- **Out of scope**: Actually rebuilding the dataset against the new vintage (handled by the dataset's own workflow).

#### Release a Dataset

- **Filename** (`<topic>_skill.md`): **`release_dataset/SKILL.md`**
- **Trigger**: When a dataset has passed UAT and is ready to become part of the canonical reference data — i.e. its slug should appear in `data/dataset-registry.json` and the `aggregations/admin/` and `aggregations/source/` packages should be rebuilt to include it.
- **Inputs the user must supply**:
  - Dataset name and slug.
  - Confirmation that UAT has passed (the skill should refuse to proceed without an explicit acknowledgement, since this gate is the only thing protecting the canonical reference tables).
- **Steps the skill performs** (numbered):
  1. Verify the dataset's outputs exist on disk under `datasets/<dataset-slug>/data/` and that `run_dataset_qc/SKILL.md` has been run successfully (e.g. by checking for a recent QC report under `datasets/<dataset-slug>/docs/qc-reports/`).
  2. Pre-flight: rebuild both aggregation entry scripts to confirm they succeed against the current registry; abort without mutating the registry if either fails (#29).
  3. Call `register_dataset(<dataset-slug>)` to write `datasets/<slug>/RELEASE.json` (per-dataset manifest, source of truth) and regenerate the derived `data/dataset-registry.json` index in one shot.
  4. Re-run `aggregations/admin/source/admin.R` and `aggregations/source/source/source.R` so the registered slug is included; surface any namespacing-bypass errors per §3.6.
  5. Compute `release_file_set(<dataset-slug>)` (manifest-derived, NOT a hardcoded list — #29) and open a release PR / commit containing exactly those paths.
- **Outputs / artifacts produced**: a new or refreshed `datasets/<slug>/RELEASE.json`; a regenerated `data/dataset-registry.json`; refreshed parquet outputs under `aggregations/admin/data/` and `aggregations/source/data/` (set is derived from aggregation specs, so it grows automatically when specs add tables); a release commit / PR containing all of the above.
- **Success criteria**: the slug's manifest is present exactly once and the derived index lists it exactly once; both aggregation builds completed without collision errors; the release PR is open and contains only the paths returned by `release_file_set()`.
- **Out of scope**: Building the dataset itself (handled by its entry-point script); creating the dataset (handled by `new_dataset/SKILL.md`); QC of the dataset (handled by `run_dataset_qc/SKILL.md`); fully transactional staged-build (carved out as a follow-up — the pre-flight in step 2 provides most of the safety today).

### 5.3 Skill conventions (Required)

Every skill file (in `.github/skills/` or
`datasets/<slug>/source/skills/`) must use the following structure.
This is the canonical template that `new_dataset/SKILL.md` and
`create_dataset_qc/SKILL.md` write into the dataset's skill folder, and
the schema that `invoke_dataset/SKILL.md` discovers against.

#### YAML frontmatter (Required)

Every skill file must begin with a YAML frontmatter block. The lint
test in `tests/test-skills.R` enforces this:

```yaml
---
filename: <basename>.md            # must equal the file's actual basename
name: <kebab-case-slug>            # short identifier
version: 1.0.0                     # semver; bump on breaking change
layout_version: v2                 # the spec version this skill targets
triggers: ["keyword 1", "keyword 2"]   # unique within scope (see below)
description: >
  One- or two-sentence summary of what the skill does.
---
```

Files in `.github/skills/_partials/` are not skills; they are shared
markdown fragments included by reference from skill bodies. Files
beginning with `_` are excluded from skill discovery and from the lint.

#### Body structure

````markdown
# <Skill Title>

- **Filename**: `<topic>_skill.md` (cross-cutting) or
  `<dataset-slug>-<topic>.md` (dataset-specific).
- **Trigger**: when the AI should invoke this skill — keyword(s) or
  user intent. Be specific enough that two skills don't both match.
  These triggers should mirror the `triggers:` array in the frontmatter.
- **Inputs the user must supply**: bullet list. Mark required vs.
  optional.
- **Steps the skill performs**: numbered list. One imperative verb per
  step.
- **Outputs / artifacts produced**: what files / branches / issues /
  PRs exist after the skill runs.
- **Success criteria**: a checklist the AI can self-verify against.
- **Out of scope**: explicit list of what this skill does **not** do,
  and which other skill handles each.

## Cross-references

If this skill produces a file that is later consumed by another skill,
cite the consuming skill by filename (e.g. "produced file is consumed
by `run_dataset_qc/SKILL.md`").

## Versioning

- Skills are versioned via the `version:` field in frontmatter (semver).
  Bump the major component on any breaking change to inputs, outputs,
  or expected side-effects.
- Breaking changes to a skill (rename, removal, or signature change of
  the inputs/outputs) require a backlog issue and a one-release
  deprecation window: keep the old filename as a thin shim that points
  to the new one.
- In-flight invocations are not migrated; the AI re-reads the skill
  file at the start of each invocation.
````

#### Owned documentation (Required when applicable)

If a skill is named in the `owner_skill:` frontmatter of a living doc under `docs/` (per the YAML frontmatter convention established by [`docs_organization/SKILL.md`](../../.github/skills/docs_organization/SKILL.md) v1.0.0), the skill is responsible for bumping that doc's `last_updated` field whenever any of its operations modify the doc. The check that owner skills exist is enforced by [`workflow_audit/SKILL.md`](../../.github/skills/workflow_audit/SKILL.md) Category 6 (BLOCKER if `owner_skill:` references a skill not in `.github/skills/`); the freshness check (`last_updated` against per-quadrant stale thresholds) is also Category 6 (WARNING).

This is a maintenance contract, not a runtime check: skills that own docs must include the bump as part of their write path. New skills filed via the §5.3 template should declare ownership explicitly under their own `## Cross-references` section ("Owned doc: `docs/<TOPIC>.md`") so the relationship is discoverable from both ends.

**Discovery rules:**

- `invoke_dataset/SKILL.md` matches against the **Trigger** field via
  keyword/intent overlap. The implementer should keep triggers
  short and disjoint.
- Triggers must be **unique within scope**: cross-cutting skills
  (`.github/skills/`) form one scope; each dataset's skill folder
  (`datasets/<slug>/source/skills/`) forms its own scope. The same
  trigger may legitimately appear in multiple per-dataset scopes
  (e.g., "QC pop-est" lives only in pop-est's scope).
- A skill must not invoke another dataset's skill directly; cross-skill
  composition is always brokered by a cross-cutting skill in
  `.github/skills/`.
